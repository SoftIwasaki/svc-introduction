// 01 — The Problem

AI開発で起きていること

💥

設計崩壊

AIが勝手にファイルを追加し、設計意図が消失する。モジュール境界が曖昧になり、保守不能なコードベースが生まれる。

🔮

暗黙の仕様補完

指示の粒度が曖昧だとAIが仕様を"推測"で埋める。なぜその実装になったか誰もわからない。

🎰

品質のギャンブル

品質はプロンプトの書き方に依存。レビュー基準が属人化し、「動いたからOK」が常態化する。

🕳️

判断の消失

設計判断の根拠が記録されない。次回も同じ問題が繰り返される。トレーサビリティ・ゼロ。

4
判断の記録なし、設計根拠なし

AIの設計判断が暗黙的。次回も同じ問題が起きる。

SVC

1
人間: 要件定義 → architecture.md承認
2
Orchestrator: バッチ分割 → Reviewer起動
3

4
Builder: テスト先行 → 自己レビュー(R-1〜R-8)
5
人間: Release Gate最終承認

各ステップに証跡。AIの判断がトレーサブル。

// 03 — Core Design

SVCの3つの核心

1. プロジェクト憲法

CLAUDE.md

AIの役割・禁止事項・承認権限を宣言的に定義。「こうしてください」ではなく「こう定義されている」と書く。

2. 3つのゲート

Design → Build → Release

AIが設計の境界を越えられないよう物理的にブロック。Hooksでファイル書き込み自体を中断。

3. 役割分離

Orchestrator / Reviewer / Builder

判断する側と作る側を分離。サブエージェントが各ルールに従い自律的に実装・レビュー。

人間は 業務判断 ・ 上位設計承認 ・ ゲートGO判定 のみに集中する。

// 04 — Cascade Generation

カスケード生成 — SVCの心臓部

ロードマップから実装指示が自動的にカスケードする。AIが「何を作るか」を考えるのではなく、構造化されたファイルから「何を作るべきか」が導出される。

🗺️

Roadmap / 要件定義

人間が業務要件と優先度を定義する。SVCの唯一の「白紙から書く」フェーズ。

architecture.md 承認 → Design Gate GO
📋

Plan File （plan.md）

ロードマップをバッチ単位に分割。各バッチの目的・スコープ・依存関係を定義。Orchestratorが自動生成。

auto-generated
Reviewer設計レビュー → Human承認 → Build Gate GO
⚡

Prompt File （prompt.md）

プランから自動導出される実装指示。Change Scope Contract・必読ファイル・絶対遵守ルールを含む。Builderはこれだけを読む。

auto-derived
Builder自律実行 → テスト → 自己レビュー → report.md
🔨

Implementation + Tests

prompt.mdのスコープ内で自律的にテスト先行実装。R-1〜R-8の自己レビューを経て report.md を出力。

autonomous
// 05 — Context Efficiency

コンテキスト分割 — 中規模開発を可能にする鍵

AIのコンテキストウィンドウは有限。SVCはカスケード構造でコンテキスト消費を最小化し、中規模開発でも自律実行を持続させる。

従来: 全文脈を毎回投入

!
プロジェクト全体をAIに説明し直す
!
コンテキストが膨張し、指示を忘れる
!
中規模以上で破綻する
📈 100%
全コンテキストを毎回消費

SVC: バッチ単位でコンテキスト分離

✓
Builderは prompt.md + guardrails だけ読む
✓
Reviewerは review.md + architecture.md だけ読む
✓
各バッチが独立。並列実行も可能
📉 ~15%
バッチ単位の必要最小限

なぜこれが効くか: 必要な役割が必要なルールだけを読む。CLAUDE.md（憲法）+ guardrails（役割別ルール）+ prompt.md（バッチ指示）の3層構造で、AIが読むべき情報を構造的に制御する。プロジェクト全体を理解し直す必要がないため、中規模開発でもトークンが枯渇せず自律実行が持続する。

// 06 — Human Role

人間は「選ぶ」だけ

設計・実装・デプロイはAIが実行する。人間の役割は「選択肢からの意思決定」に集約される。

🧭

設計判断

Orchestratorがトリアージし、選択肢を提示。人間は選ぶだけ。判断はDECISIONS.mdに自動記録される。

## トリアージ結果 選択肢: A: BIPと相乗り ($0増) B: 独立Plan (+$13/月) 推奨: A
✅

ゲート承認

Reviewerのレビュー結果を確認し、GO / NO GO を判定。承認はreview.jsonに証跡として永続化。

GO
🚀

リリース判定

全テスト結果・Reviewer承認・品質指標を確認し、最終GOを出す。ここだけは全レベルでHuman必須。

BE 147 + FE 131 = 278 PASS

結果: 人間は毎回のコード全行レビューから解放され、業務上の意思決定に集中できる。品質基準はguardrailsに明文化済み。設計判断の経緯はDECISIONS.mdで永続化。「前もこの指示したよね」がなくなる。

// 07 — External AI Review

/sync-pack — 判断ボトルネックの解消

人間が判断できないとき・迷うとき、プロジェクトの全コンテキストをパッケージ化して外部AIに投げられる。

/sync-pack
→
1つのMarkdownに自動集約

外部AI (ChatGPT等)

異なるモデル・異なる視点で「第三者の目」レビュー

参考水準

# 自動集約されるコンテキスト（約2000行） ├── プロジェクト憲法（CLAUDE.md全文） ├── ゲート状態サマリ（design / build / release） ├── 意思決定履歴（DECISIONS.md） ├── プラン・レビュー状況（plan.md + review.json） ├── 基本設計（architecture.md） └── レビュー依頼テンプレート（4観点）

// 10 — Role Separation

役割分離 — サブエージェントアーキテクチャ

Orchestrator

進行管理・トリアージ・ゲート更新 — 設計・実装は一切しない

┃ サブエージェント起動 ┃

Reviewer

設計整合性・セキュリティ
非機能審査・R-1〜R-8

reviewer.md

Builder

テスト先行実装
自己レビュー・report出力

builder_*.md

Librarian

SSOT正本管理
知識配給・乖離検知

librarian.md

各サブエージェントは自分のガードレールファイルだけを読む — コンテキスト分離の実践。
Orchestratorが「設計・実装を一切しない」のが設計上の要点。AI同士の馴れ合いを防止。

// 11 — Batch Structure

バッチ構造と Change Scope Contract

LIVE DATA — PLAN-0001-csv-upload
plans/PLAN-0001-CSV取込/ ├── plan.md ← バッチ分割の根拠 ├── review.json ← 全承認記録 ├── batch1/ │ ├── prompt.md ← 実装指示 │ ├── review.md ← Reviewerレビュー │ └── report.md ← Builder報告 ├── batch2/ └── batch3/

Change Scope Contract

prompt.mdに変更スコープを明示。AIが勝手にスコープを広げることを構造的に防ぐ。

### Change Scope Contract Allowed: models.py, admin.py, tests/test_models.py Conditional: settings.py （理由あれば変更可、report.md記載必須） Forbidden: urls.py, views.py （batch3の対象。触るならESCALATE）

// 13 — Control Modes

リスクに応じた3つの統制モード

Full SVC

DBスキーマ・API契約・権限制御

Design Gate + Build Gate + Reviewer + Human承認 + report.md

Lite SVC

機能内の局所的な変更

prompt.md + 自己レビュー + report.md

Fast Lane

リネーム・コメント修正等

差分レビューのみ

判定基準（いずれか1つでもYes → Full SVC）

影響軸具体例
データ影響DBスキーマ / migration に影響
公開契約影響APIリクエスト / レスポンスに影響
権限制御影響認可 / ロール / 表示制御に影響
横断影響複数モジュール / shared / infraに波及

// 17 — Immutable Principles

Human Judgment Only

Gated Orchestration

Test-First + Self-Verification
