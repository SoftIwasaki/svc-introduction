// 01 — The Problem & Why SVC

AIに「お願い」しても品質は保証されない

設計崩壊・暗黙の仕様補完

AIが勝手にファイルを追加し設計意図が消失する。指示が曖昧だとAIが仕様を"推測"で埋め、なぜその実装になったか誰もわからない。

品質のギャンブル・判断の消失

品質はプロンプトの書き方に依存し「動いたからOK」が常態化。設計判断の根拠が記録されず、次回も同じ問題が繰り返される。

SVCはこの問題を3つの仕組みで解決する：
プロジェクト憲法（CLAUDE.md） で宣言的にAIを制御、 3つのゲート で物理的に暴走をブロック、
カスケード生成 でロードマップから実装指示を自動導出する。

// 02 — The Vision

// 04 — Context Efficiency

コンテキスト分離 — 中規模開発を可能にする鍵

AIのコンテキストウィンドウは有限。カスケード構造がコンテキスト消費を最小化し、中規模でも自律実行を持続させる。

従来: 全文脈を毎回投入

!
プロジェクト全体をAIに説明し直す
!
コンテキストが膨張し、指示を忘れる
!
中規模以上で破綻する
100%
全コンテキストを毎回消費

SVC: バッチ単位でコンテキスト分離

✓
Builderは prompt.md + guardrails だけ読む
✓
Reviewerは architecture.md + review.md だけ読む
✓
各バッチが独立。並列実行も可能
~15%
バッチ単位の必要最小限

なぜ中規模開発で効くか: BIP Cost Insightは設計文書71本・画面仕様40本・API仕様60エンドポイントの中規模システムだが、Builderが1バッチで読むのはCLAUDE.md + guardrails + prompt.md の3ファイルだけ。プロジェクト全体を理解し直す必要がないため、22バッチを逐次実行してもトークンが枯渇せず自律実行が持続する。

// 05 — Human Role

人間は「選ぶ」だけ

設計・実装・テスト・レビューはAIが実行する。人間の役割は「Orchestratorが提示する選択肢から選ぶ」ことに集約される。

トリアージ → 選択 → 自動記録

## トリアージ結果 判定: Clarification 理由: Azure App Service Planの配置戦略 選択肢: A: BIPと相乗り ($0増) B: 独立Plan (+$13/月) 推奨: A 不確実性: CPU負荷は実測後再判断 ─── 人間が「A」と返すだけ ─── # → DECISION-0007 として自動記録 decision: "BIPと相乗り" rationale: "月額増分$0" revisit: "CPU 70%超が1週間継続なら"

人間に求められること

業務判断の選択

「A or B？」に答えるだけ。技術的判断はOrchestratorが選択肢に含めて提示する。

ゲートGO判定

Reviewerのレビュー結果を見て GO / NO GO。承認はreview.jsonに証跡として永続化。

Release Gate最終承認

全テスト結果と品質指標を確認。ここだけは委譲レベルに関わらずHuman必須。

// 08 — Control Levels

統制の段階性 — リスクに応じて強度を変える

すべての変更を同じ強度で扱わない。影響の大きさでモードを選択し、委譲マトリクスでAIの自律範囲を制御する。

3つの統制モード

Full SVC

DBスキーマ・API契約・権限制御に影響する変更
→ Design Gate + Build Gate + Reviewer + Human承認

Lite SVC

機能内の局所的な変更
→ prompt.md + 自己レビュー + report.md

Fast Lane

リネーム・コメント修正等
→ 差分レビューのみ。gate不要

承認委譲マトリクス

承認対象L1 StrictL2L3
Design GateHHAI
Build Gate (Full)HHAI
Build Gate (Lite)HAIAI
上位設計遡及HHH
Release GateHHH

安全弁: 上位設計遡及とRelease Gateは全レベルでHuman必須。設計の根幹とデプロイ判定は絶対に委譲しない。

// 10 — Immutable Principles

不変の原則

01

Plan First

実装より計画を優先する

02

Human Judgment Only

最終判断は人間のみ。ただし「選ぶだけ」でよい構造を作る

03

Gated Orchestration

3つのゲートでAIの暴走を機械的に防ぐ

04

Cascade Generation

ロードマップから実装指示を自動導出する

05

Context Isolation

バッチ単位でコンテキストを分離し、中規模でも持続可能にする

06

Traceable Decisions

すべての判断を記録する。トレーサビリティ・ゼロを許さない

07

CLAUDE.md = Constitution

すべての役割はプロジェクト憲法に従う

SVCは「AIを信頼しない」フレームワークではない。
「AIを信頼できる構造を作る」 フレームワークだ。
